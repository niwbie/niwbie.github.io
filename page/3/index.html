格式化字符串漏洞的利用

我们知道格式化字符串漏洞可以向内存中写入数据，那么要如何利用这个漏洞获得shell呢？
这一节主要熟悉一下pwntools的使用。

首先我们来看源程序：

gcc -m32 -fno-protector -O0 fmttest.c -o fmttest
程序编译时开启了栈保护。

在这个程序中，我们为了实验方便，循环读取输入，然后使用printf输出。
很明显可以看出这里存在格式化字符串漏洞。那么为了写入shell，我们的思路是将printf函数的地址改成system函数的地址，这样当我们再次输入”/bin/sh”时，就可以获得shell了。

先说一个知识点，就是格式化字符串中的”$”
一般情况下，如果我们的字符串为”%x”，就会将取1个参数，那么如果是”%3$x”，则是取出第三个参数。即利用”$”可以取出任意位置的参数。利用这点，可以做很多是，比如dump处程序的bin文件。

当”$”取出的第N个参数刚好是该字符串的地址是，我们称N为offset。
比如当 N=4 时，我们可以构造这样一个payload： addr + “%4$s” ，那么printf就会打印出addr指向的内存的字符串，知道\0为止。这样就是所谓的内存泄露。比如我们利用这样特性， 将addr构造成print函数的地址（print_got）,那么我们就讲得到printf函数在程序中的地址了。

Windows下的ROP绕过DEP保护

这篇介绍经典的DEP绕过方法：通过VirtualProtect绕过DEP。
test.exe

DEP保护的原理，就是将数据所在内存页标识为不可执行，当程序溢出成功转入shellcode时，程序会尝试在数据页面上执行指令，而有了DEP，此时CPU会抛出异常，而不是去执行指令

如果我们让程序跳转到一个已经存在的系统函数中去呢？这时的DEP是不会拦截的，也就有了我们的ROP攻击。
简而言之，只要为shellcode中的每条指令都在代码区找到替代指令，就可以完成我们的功能了。但是这种操作难度极大，并不是每条指令都能找到，所以就有了以下三种折中的办法:

通过跳转到ZwSetInformationProcess函数将DEP关闭后再转入shellcode执行
通过跳转到VirtualProtect函数来讲shellcode所在内存页设置为可执行状态，然后再转入shellcode执行。
通过跳转到VirtualAllloc函数开辟一段具有执行权限的内存空间，然后将shellcode复制到这段内存中执行。

这一节根据实例，介绍通过方法2来绕过DEP。

先来看VirtualProtect函数：

BOOL VirtualProtect{
LPVOID lpAddress,
DWORD dwsize,
DWORD flNewProtect,
PDWORD lpflOldProtect
}

lpAddress:内存起始地址
dwsize:内存区域大小
flNewProtect:内存属性，PAGE_EXECUTE_READWRITE(0x40)
lpflOldProtect:内存原始属性保存地址
